# Niche Cases & Edge Scenarios: High-Concurrency Ticketing Systems

In a mission-critical ticketing system (e.g., Coldplay tour, FIFA World Cup), the difference between a "working" app and a "Senior-grade" app lies in how it handles the 1% of edge cases. Below is an analysis of niche scenarios and their architectural solutions.

## 1. The "Ghost Seat" Problem (Reservation Expiration)
**The Case:** A user selects 4 seats and proceeds to checkout. They get distracted and leave their browser open for 30 minutes.
- **The Risk:** Those seats are locked/reserved but never paid for, preventing other users from buying them.
- **The Solution:** 
    - **TTL (Time-To-Live) Reservations:** Store reservations in Redis with a 5-10 minute TTL.
    - **Cron/Worker cleanup:** A background worker (BullMQ) that automatically reverts `SOLD` or `RESERVED` status to `AVAILABLE` if no payment confirmation is received within the window.

## 2. Distributed Transaction "Partial Failure"
**The Case:** The system decrements the ticket stock in the DB, but the connection to the Payment Gateway drops, or the server crashes before creating the Order record.
- **The Risk:** Stock is lost (ghost decrement), but no order exists.
- **The Solution:** 
    - **Saga Pattern / Two-Phase Commit:** Wrap the stock decrement and payment initiation in a state machine.
    - **Reconciliation Service:** A nightly job that compares `sum(orders.quantity)` vs `ticket.initial_quantity - ticket.remaining_quantity` and adjusts for discrepancies.

## 3. The "Double-Click" (Idempotency)
**The Case:** A user with a laggy connection clicks "Purchase" multiple times.
- **The Risk:** Multiple orders created for the same user, or double-charging.
- **The Solution:** 
    - **Idempotency Keys:** Require a unique `request_id` (generated by the frontend per transaction attempt).
    - **Redis Locking:** Before processing, check `EXISTS order_lock:{idempotency_key}`. If exists, reject the duplicate.

## 4. Ticketing Bots & Scalping Surges
**The Case:** Scalper bots attempt to buy 500 tickets in the first 200ms of the sale.
- **The Risk:** Genuine fans are locked out; server crashes due to the sudden spike.
- **The Solution:** 
    - **Proof-of-Work / CAPTCHA:** Integrate Turnstile or reCAPTCHA v3 on the "Buy" button.
    - **Dynamic Throttling:** Increase the "Waiting Room" strictness based on the rate of `403 Forbidden` errors detected.

## 5. Clock Skew & "Pre-emptive" Hits
**The Case:** An event starts at 10:00:00 AM. Users in different regions have slightly different system clocks.
- **The Risk:** Some users access the "Buy" button 2 seconds early, others 2 seconds late.
- **The Solution:** 
    - **Server-Side Gatekeeping:** Use the server's NTP-synchronized clock as the absolute truth.
    - **Jitter Handling:** Intentionally add a random delay (100-500ms) to the "Sale Start" to prevent a perfect "thundering herd" at exactly 0ms.

## 6. Seat Proximity & Social Distancing
**The Case:** Certain events require leaving a gap between occupied seats (e.g., during health restrictions) or preventing "single seat orphans" (leaving one empty seat between two bookings).
- **The Risk:** Fragmented seating that cannot be sold as a block.
- **The Solution:** 
    - **Algorithm-Based Allocation:** Don't just pick the "next available" ID. Use a seating algorithm that prioritizes filling rows or maintaining contiguous blocks.

## 7. The "Refund Storm" (Event Cancellation)
**The Case:** A major event is cancelled 1 hour before starting.
- **The Risk:** 50,000 users requesting refunds simultaneously, overwhelming the Payment Gateway API and our DB.
- **The Solution:** 
    - **Bulk Status Update:** Change event status to `CANCELLED` and stop all sales via cache instantly.
    - **Queue-Based Refunds:** Don't process refunds synchronously. Push all orders into a "Refund Queue" and process them at a rate accepted by the payment provider (e.g., Stripe's rate limits).

## 8. Database "Hot Keys"
**The Case:** 100,000 users are all hitting the *exact same* `Ticket` record in PostgreSQL.
- **The Risk:** Row-level lock contention in the DB makes the app crawl, even if CPU is low.
- **The Solution:** 
    - **Redis-Based Inventory Counter:** Use Redis `DECR` (which is atomic and extremely fast) for the first line of inventory tracking, then sync to the DB asynchronously or in smaller batches.
